# Conductor Vision

Conductor should become something rarer than an AI music app.

It should become a new instrument category.

Not "music generation software."
Not "live coding with a nicer UI."
Not "Ableton for AI."

Conductor should feel like a living ensemble you can play with your hands, your timing, and your taste.

This is a high bar. It asks more of the product than novelty. It asks more than fast generation, more than a compelling demo, more than a pile of clever controls. It asks for an instrument that can support practice, style, risk, and performance. It asks for a system that is not exhausted after five minutes. It asks for something people can love enough to develop taste inside of.

That is the ambition worth pursuing.

There are already many ways to make music with software. There are already many ways to ask AI for audio, stems, loops, and ideas. Conductor should not matter because it is another entry in that list. It should matter because it offers a distinct way of relating to music creation: not as assembly, not as editing, not as prompting, but as conduct. The performer should feel less like an operator and more like someone shaping weather, attention, energy, and behavior inside a living musical field.

## The Vision

When you open Conductor, music is already alive.

Not a blank canvas.
Not a prompt input.
Not a code editor waiting to be fed.

There is already a pulse.
There is already a system breathing.
There is already an ensemble holding tension.

Your role is not to author every atom.
Your role is to enter the flow and shape it.

Conductor should be an instrument that begins in motion and asks for conduct, not setup.

That single inversion is foundational. Most software begins by asking the user to define. Pick a project. Choose a tempo. Select a sound. Enter a prompt. Add a track. Conductor should begin by offering a situation. Here is a groove. Here is a tension. Here is an unstable equilibrium. What will you do with it? The product should invite intervention into an existing current, because intervention is where performance begins.

## The Core Experience

Conductor should feel like standing in front of a strange, intelligent band.

The kick is stubborn.
The hats are twitchy.
The bass is heavy and responsive.
The texture is atmospheric and unruly.
The lead is asleep until invited.

Each voice should have:

- identity
- inertia
- memory
- a tendency to behave a certain way

The performer should learn how to work with those tendencies.

This is important. The system should not feel like a blank machine that obediently accepts arbitrary requests. It should feel like something with grain. Some voices should cooperate easily. Others should be difficult, volatile, or heavy. One lane might lock beautifully when held steady. Another might become brittle if pushed too far. Another might start dull and become extraordinary only after careful shaping. This friction is not a flaw. It is what gives the instrument texture, and what gives the performer something to learn.

Not:

- change parameter 4
- make it darker
- send a better prompt

But:

- tighten the drummer
- starve the bass, then let it bloom
- hold the groove while destabilizing the air
- wake the lead for one phrase, then kill it
- force the ensemble through a narrow gate, then let it explode open

The difference between these two lists is the difference between software and instrument. Software asks for outcomes. Instruments invite moves. In a great instrument, the player begins to think in verbs, in rhythms, in relationships, in pressure and release. They do not merely seek a result. They perform a transformation.

## The Interface

Conductor should feel like a living score.

A full-screen stage of voices.
No heavy SaaS chrome.
No inspector panels trying to explain everything.
No DAW cosplay.

Just the ensemble, visibly alive.

The screen should not feel like a productivity application arranged around panels and controls. It should feel like a stage, a score, a machine in a state of tension. It should contain enough information to orient the performer, but never so much that it loses its atmosphere. Space, rhythm, contrast, and motion should do as much explanatory work as labels. The interface should teach through presence before it teaches through text.

Each lane should feel almost biological:

- pulsing
- thickening
- thinning
- locking in
- fraying
- glowing when energized
- going pale when stripped

The screen should make musical state visible.

At a glance, the performer and audience should be able to tell:

- who has power
- who is restrained
- who is mutating
- where tension is building
- what is about to land

The performer should not need to translate the screen into musical understanding. The screen itself should already be musical understanding. If the bass is carrying power, that should be visible. If the hats are becoming agitated, that should be visible. If the texture is blooming into the vacuum left by stripped drums, that should be visible. The UI should not merely annotate events. It should render the current condition of the ensemble.

## The Instrument

The lanes are the instrument.

Not the prompt box.
Not the code editor.
Not a bank of abstract adjective buttons.

Each lane should be a playable musical voice.

This point should be taken literally. The lane is not just a row in a mixer and not just a collapsed code block. It is the place where the performer touches the music. The lane should carry pulse, pressure, identity, responsiveness, and consequence. It should be possible to develop touch on a lane. It should be possible to know that a great performer can get more out of the same lane than a beginner can.

The performer should be able to:

- focus a lane
- reshape its pulse
- push it into change
- pin it in place
- strip it back
- wake it up
- destabilize it
- recover it

The lane is not just a track.
It is a living voice with behavior.

That means lane design matters more than almost anything else in the product. The exact way a lane responds to focus, drag, hold, mutation, and timing will define whether Conductor becomes memorable. The lane interaction model is not a detail. It is the instrument.

## The Skill

Conductor should reward technique.

Not fake technique based on navigating menus.
Not generic creativity based on typing prompts.

Real technique based on:

- attention
- timing
- restraint
- sequencing
- recovery
- reading the ensemble
- understanding inertia

The instrument should support recognizable moves.

This is one of the clearest tests for whether the product is real. Can players talk to each other in terms of moves? Can they teach each other? Can they watch a performance and recognize good judgment? Can they see someone recover from over-pushing the ensemble and think, that was skillful? If the answer is yes, Conductor starts crossing the line from novelty into culture.

Examples:

- isolate the bass, mutate it, then drop the kick back in on the phrase
- freeze the groove, strip texture, build hats, then release
- starve everything except percussion, then let the pad bloom

If performers can develop signature moves, the instrument has soul.

What matters is not just that the interface allows action. It must allow style. A serious instrument lets people become distinct from one another. Two pianists playing the same notes can still sound different. Two DJs using the same tracks can still feel different. Conductor should aim for that same possibility. Users should be able to develop recognizable habits, phrasing, patience, aggression, fragility, and dramatic sense.

## Semi-Living Voices

This is the most important leap.

The voices should not feel like dead rows with parameters.

They should feel semi-alive.

Not fully autonomous.
Not random for its own sake.

But behavior-rich.

For example:

- if a lane is pushed too hard, it becomes unstable
- if a lane is held steady, it settles into identity
- if the groove is frozen, voices remain rhythmically loyal while timbre evolves
- if most of the ensemble is stripped, a texture lane may bloom to fill the void
- if kick and bass are tightly coupled, changing one changes how the other feels

The performer is not operating a spreadsheet.
They are shaping a responsive musical ecology.

This is the center of the whole vision.

The reason most generative systems feel shallow is that they behave like vending machines or randomizers. You ask, they emit. Or you perturb, and they jump somewhere else. That can be entertaining, but it does not sustain practice. A responsive ecology is different. In an ecology, everything has tendencies, dependencies, and thresholds. Things can be coaxed, cornered, destabilized, or calmed. A performer can develop intuition for the whole system, not just issue commands into it.

If Conductor gets this right, the music will feel less like generation and more like relationship.

## The Mental Model

Conductor is a live instrument for shaping the behavior of musical voices over time.

That is the foundation.

It should support:

- beginners using mute, solo, focus, and mutate
- intermediate players learning timing and recovery
- advanced players mastering inter-voice relationships and phrase stacking

This sentence should be strong enough to hold the whole product. It should inform tutorials, interface decisions, naming, onboarding, performance design, and future features. If a new idea does not strengthen this mental model, it is probably a distraction.

## Gesture

Lane interaction should feel tactile.

Not just click and button press.

The performer should be able to:

- drag across a pulse strip to reshape density
- press and hold a lane to pin it
- flick a lane upward to wake it
- pull it downward to strip it
- sweep across several lanes to apply ensemble pressure
- feel phrase arrivals land with visible consequence

The gestures themselves should become technique.

Tactility is not only about physical realism. It is about consequence. The performer should feel that their touch changes the state of the music in a meaningful way. A quick flick and a held pressure should not mean the same thing. A gesture made in the middle of a phrase should feel different from one landed on the bar. A sweep across the ensemble should feel categorically different from a precise local intervention. If the gestures have weight, timing, and differentiated outcomes, players will start to form habits of touch. That is where instrument literacy begins.

## Stagecraft

Conductor should be watchable.

Not just usable.
Watchable.

Even without sound, the screen should carry tension.

That means:

- changes should visibly gather before landing
- phrase boundaries should feel ceremonial
- cues should leave visible traces
- the focused lane should feel charged
- the ensemble should breathe as energy rises and falls

The product lives partly in the space of:

- instrument
- performance
- visual theater
- composition
- software

This matters more than it might seem. Watchability is not just a nice property for marketing videos. It is part of the product itself. A watchable instrument creates social proof, tutorial energy, streamability, and cultural meaning. It lets people understand that something skillful is happening even before they know the details. If the interface cannot carry dramatic information visually, the performance loses an entire dimension of its power.

## Emotional Arc

A great Conductor session should feel like:

- entering an existing pulse
- probing and listening
- finding which voices are alive
- making small moves
- discovering resistance
- learning the ensemble
- creating tension
- overpushing something
- recovering
- finding a moment worth saving
- building a set out of moments
- performing the system instead of merely using it

This arc is the emotional promise of the product. The user should not feel like they are endlessly auditioning generated variations. They should feel like they are entering into relation with a system, discovering its edges, shaping it, losing control of it, regaining control, and finally finding themselves able to perform through it. The product should support not just output, but a sense of journey.

## What To Avoid

Conductor should avoid collapsing into generic software.

That means avoiding:

- too many utility panels
- global adjective shortcuts as the main mechanic
- too much visible AI "generate" framing
- DAW mimicry
- over-explaining the engine
- identical generic controls on every lane
- reducing the whole product to an XY pad

These things reduce mystery, flatten the skill ceiling, and weaken the instrument feel.

Conductor should be careful not to explain itself to death. The current generation of creative tools often compensates for weak interaction by adding more language, more panels, more helper text, and more visible machinery. That is understandable, but it is not the path to an instrument. A good instrument contains a productive amount of opacity. It reveals itself through use. It gives enough footholds to begin, while leaving enough depth to discover.

## What Could Make It Great

Three things matter most:

### 1. Lane personality

Each voice should have its own expressive identity and signature controls.

### 2. Behavioral depth

The system should have inertia, not instant obedience.

That gives the performer something to learn.

### 3. Phrase ritual

Changes should land with visible and satisfying consequence.

That is where performance becomes performance.

Phrase ritual deserves emphasis. Every serious music system has some version of ceremony around change: the downbeat, the drop, the entrance, the cut, the return. Conductor should make these moments feel tangible. A change should not merely occur. It should arrive. The performer should anticipate it. The audience should sense it. The interface should acknowledge it. This is one of the most powerful opportunities in the whole product, because it turns state change into drama.

And if there is one more ingredient beyond these three, it is danger. The system should not collapse into chaos, but it should allow the performer to flirt with instability. Great performances are compelling because they sit near the edge of failure and control. If Conductor feels too safe, it will feel like software. If it allows calculated risk, it can start to feel alive.

## The Future

In the best version, people talk about Conductor the way they talk about instruments:

- "She is amazing at controlling tension in it."
- "He has crazy lane timing."
- "Their bass moves are unmistakable."
- "You can always tell when someone really knows how to play it."

That is the goal.

Not:

- it generated something cool

But:

- they played it beautifully

That is the real north star.

The highest compliment for Conductor should never be that it is smart, fast, or technically impressive. Those things may matter, but they are not the point. The highest compliment should be that someone used it with taste, with nerve, with timing, with style. The system should make room for beauty, surprise, and authorship to appear through performance.

If Conductor succeeds, people will not merely use it to make music. They will use it to become a kind of musician that did not quite exist before.

## Concrete Product Shape

The vision needs a concrete shape in order to stay honest.

Conductor should not try to do everything at once. It should not attempt to immediately become a full DAW, a complete live-coding environment, a social network for performances, and a generalized AI music workstation. The right path is to build a strong instrument core and let the rest of the product grow around it.

That core should be:

- a living ensemble of voice lanes
- one focused lane at a time
- phrase-aware interaction
- a small set of meaningful lane verbs
- moment capture and recall
- visible musical state

Everything else should be in service of that.

## The Voice Model

The most important product object is the voice.

A voice is not just a track and not just a chunk of code. It is a musical actor with a role in the ensemble.

Each voice should have:

- a role
- a pulse representation
- a state
- a behavior profile
- a small number of expressive controls
- a code backing
- a history of recent transformations

### Example voice roles

- kick
- snare or clap
- hats
- percussion
- bass
- chords
- lead
- texture
- vocal fragment
- transition or impact lane

The app does not need all of these at once. In fact, it should probably start with fewer. But the model should allow them.

### Example visible voice states

- sleeping
- active
- focused
- pinned
- mutating
- unstable
- recovering
- soloed
- suppressed
- blooming

These states are useful because they give the UI a language that is not purely technical. A lane is not merely "on" or "off." It is in a musical condition.

### Example behavior profiles

A kick lane might be:

- stable
- highly phrase-sensitive
- low melodic variance
- high rhythmic authority

A texture lane might be:

- loose
- slow to settle
- highly responsive to space and tone
- willing to bloom when other lanes are stripped

A bass lane might be:

- groove-coupled to the kick
- sensitive to pressure
- capable of mutation without losing role identity

These profiles can start simple and become richer over time.

## Lane Anatomy

Every lane should have a clear visual and interaction anatomy.

### Compact lane

The compact lane is the default state for most voices.

It should show:

- role label
- short descriptor
- pulse strip
- mute
- solo
- whether the lane has something queued
- whether the lane is in a special state such as pinned or unstable

The compact lane should not try to show every detail. Its job is to keep the ensemble readable.

### Focused lane

The focused lane is the active performance surface.

It should expand to reveal:

- enlarged pulse strip
- drag zones
- one or two signature controls
- one or two generic controls
- lane verbs
- quantize target
- pending actions
- code texture
- optional natural-language command input scoped to the lane

The focused lane is where the product either becomes an instrument or fails to.

## Signature Controls

One mistake would be giving every lane the same four generic sliders forever.

Instead, each lane should have one signature expressive control that reinforces its role.

Examples:

- kick: `weight`
- hats: `chatter`
- bass: `pressure`
- chords: `spread`
- lead: `reach`
- texture: `bloom`
- percussion: `scatter`

These controls should not just be cosmetic labels. They should map to meaningful parameter bundles under the hood.

For example, `pressure` on bass might influence:

- gain range
- filter envelope
- note density
- distortion tendency
- low-pass target

This is how the interface stays poetic while remaining technically grounded.

## Lane Verbs

Conductor should build a recognizable vocabulary of lane verbs.

These should be:

- short
- musical
- composable
- learnable
- visually legible

### Candidate core lane verbs

- `mutate`
- `strip`
- `double`
- `halve`
- `twist`
- `tighten`
- `open`
- `scatter`
- `wake`
- `pin`

### What these verbs might mean

`mutate`
: produce a lane-scoped variation while preserving role identity

`strip`
: reduce density, ornament, and harmonic or timbral complexity

`double`
: increase event frequency or subdivision pressure

`halve`
: reduce event frequency or simplify subdivision

`twist`
: change pattern logic or articulation without fully changing role

`tighten`
: reduce rhythmic looseness, shorten tails, increase cut behavior

`open`
: expand spectral or spatial footprint

`scatter`
: introduce controlled irregularity or distributed accents

`wake`
: bring a sleeping lane into activity

`pin`
: temporarily resist mutation and preserve current identity

These definitions should be stable enough to teach but flexible enough to remain expressive.

## Ensemble Moves

Lane verbs are the local vocabulary.
Ensemble moves are the large-scale vocabulary.

The best ensemble moves should affect multiple voices in role-aware ways.

### Candidate ensemble moves

- `build`
- `drop`
- `freeze groove`
- `release`
- `narrow gate`
- `blackout`
- `bloom`
- `swap weight`

### Example meanings

`build`
: raise tension across active lanes, often through density, filter movement, or anticipation

`drop`
: create a visible arrival and reassert core rhythmic force

`freeze groove`
: preserve rhythmic identity while allowing timbral evolution

`release`
: reduce accumulated pressure and open breathing room

`narrow gate`
: constrain spectral and rhythmic width across the ensemble

`blackout`
: abruptly suppress most voices, leaving one or two exposed

`bloom`
: allow harmonic and atmospheric voices to widen and fill space

`swap weight`
: shift perceived emphasis from one family of voices to another

These are not just features. They are potential stage moves.

## Phrase System

Phrase interaction is a technical requirement, not a nice-to-have.

Without phrase-aware interaction, the system will feel reactive rather than musical.

Conductor should let the performer land actions at musically meaningful boundaries:

- now
- next beat
- next bar
- next 2 bars
- next 4 bars
- next 8 bars

The phrase system should support:

- quantized triggering
- visible queues
- stacked pending actions
- cancellation before landing
- visual anticipation of arrival

This is where timing becomes visible skill.

## Moments

Moments are snapshots of musically meaningful states.

A moment should be more than a preset and less than a full project file.

It should capture:

- the current voice set
- lane states
- essential code
- macro conditions
- queued or recent transformations
- musical tags or descriptors

Examples:

- "thin pressure"
- "drop held too long"
- "metal hats with ghost bass"

Moments matter because they turn happy accidents into playable memory.

They also support the emotional arc of a set:

- find something
- save it
- leave it
- come back transformed

## Two Modes of Intelligence

Conductor should blend two kinds of intelligence.

### 1. Deterministic local intelligence

This includes things like:

- muting
- soloing
- gain changes
- density drag
- parameter macro mapping
- phrase quantization
- state transitions

These should feel immediate and trustworthy.

### 2. Generative model intelligence

This includes things like:

- lane mutation
- ensemble reshaping
- natural-language direction
- new voice generation
- recovery from bad states
- continuity-aware variations

This should feel expressive, but constrained.

The product gets much stronger if users can trust the local layer and selectively invoke the generative layer.

## Technical Foundations

A strong vision needs a strong substrate.

Conductor should rest on a technical architecture that preserves immediacy while allowing controlled intelligence.

### 1. Canonical musical state

There should be a canonical in-memory state model that represents the active ensemble.

That model might include:

- voices
- voice roles
- voice states
- current code per voice
- voice macro values
- phrase settings
- queues
- moment history
- ensemble state

The UI should derive from this model, and the audio engine should be re-rendered from it.

### 2. Voice parser and reconstructor

Because Strudel code remains the truth layer underneath, Conductor needs reliable parsing and reconstruction.

That layer should support:

- separating voices
- preserving comments and labels
- updating voice-local transformations
- rebuilding full code from structured voice state

This already exists in partial form and should become more central.

### 3. Macro mapping layer

There should be a layer that maps poetic controls to concrete code changes.

For example:

- `pressure` on bass
- `chatter` on hats
- `bloom` on texture

Each macro should map to a bounded transformation bundle instead of a single raw parameter.

This lets the UI stay expressive while the engine stays coherent.

### 4. Quantized action queue

Actions should not always execute immediately.

The system should maintain a queue of pending actions with:

- target voice or ensemble
- action type
- parameters
- landing boundary
- status

That queue should drive both the UI and the eventual code updates.

### 5. Behavioral state engine

To create semi-living voices, Conductor will likely need a lightweight behavior engine.

This does not need to be complex at first.

A lane could track values like:

- stability
- agitation
- density
- presence
- coupling strength
- recent mutation count

Those values can then influence how a lane responds to future actions.

This is one of the most important technical ideas in the whole product.

### 6. Prompt scoping system

When the model is invoked, prompts should be tightly scoped.

Examples:

- mutate this bass voice, preserve rhythm and role
- add a sleeping texture lane in the current tonal space
- make the active hats more brittle without touching other voices
- freeze groove, but evolve timbre across the ensemble

This will be much stronger than sending broad full-song prompts every time.

### 7. Moment store

Moments should be stored as structured data, not just opaque text blobs.

That allows:

- recall
- comparison
- branching
- future arrangement tools
- shareable states

## A Possible Data Model

The exact types can change, but the shape should trend toward something like:

```ts
type VoiceRole =
  | 'kick'
  | 'hats'
  | 'bass'
  | 'texture'
  | 'lead'
  | 'perc'
  | 'chords';

type VoiceState =
  | 'sleeping'
  | 'active'
  | 'focused'
  | 'pinned'
  | 'mutating'
  | 'unstable'
  | 'recovering'
  | 'soloed';

interface VoiceBehavior {
  stability: number;
  agitation: number;
  coupling: number;
  mutationLoad: number;
}

interface VoiceMacros {
  level: number;
  density: number;
  signature: number;
  tone: number;
}

interface Voice {
  id: string;
  role: VoiceRole;
  label: string;
  code: string;
  state: VoiceState;
  muted: boolean;
  soloed: boolean;
  macros: VoiceMacros;
  behavior: VoiceBehavior;
}

interface QueuedAction {
  id: string;
  target: 'voice' | 'ensemble';
  targetId?: string;
  action: string;
  landing: 'now' | 'beat' | 'bar' | '2bars' | '4bars' | '8bars';
  status: 'queued' | 'landed' | 'cancelled';
}
```

This is not final. It is a conceptual anchor.

## Implementation Philosophy

The product should be built in layers.

### Layer 1: trustworthy local play

First make sure the player can:

- focus lanes
- mute and solo reliably
- drag density
- adjust signature controls
- queue actions on phrase boundaries

If this layer does not feel good, no amount of AI will save the product.

### Layer 2: lane-scoped intelligence

Then make sure lane verbs work well:

- mutate
- strip
- double
- halve
- twist

These should be constrained, role-aware, and recoverable.

### Layer 3: ensemble intelligence

After that, add:

- build
- drop
- freeze groove
- release

This is where larger-scale performance becomes viable.

### Layer 4: behavior and ecology

Then deepen the product by adding:

- lane inertia
- response curves
- inter-lane coupling
- blooming and suppression states
- instability and recovery

This is what makes Conductor feel alive.

### Layer 5: memory and authorship

Finally, strengthen:

- moments
- branching
- saved sets
- tutorialized moves
- performance recall

This is what turns a cool tool into a durable practice.

## A Standard To Hold

Conductor should be judged against a hard standard.

Would a great performer still be interesting on this system after one hundred hours?

If the answer is no, the product is still too shallow.

Would two skilled users feel recognizably different from each other?

If the answer is no, the product still lacks expressive range.

Would a viewer be able to sense intention, tension, and release just by watching the screen?

If the answer is no, the interface still lacks stagecraft.

Those are the standards worth building toward.

## Screen Architecture

The screen should be organized around performance, not administration.

At the highest level, the interface should be composed of five visible zones:

### 1. Transport and phrase header

This zone should show:

- play or stop state
- BPM
- phrase position
- current quantize target
- whether anything important is queued

This header should be compact. It is not the star of the show. Its job is to orient the performer and the audience.

### 2. Live score

This is the heart of the product.

The live score is the stack of voice lanes. It should dominate the screen. It should feel like the stage itself.

Each lane should communicate:

- role
- current energy
- pulse pattern
- state
- muting or soloing
- whether something is pending

The live score should be readable at a glance from several feet away.

### 3. Focus surface

When a lane is focused, it should expand without breaking the flow of the score.

The focused area should reveal:

- a larger pulse strip
- signature control
- local controls
- lane verbs
- pending queue
- lane-local command input

The focused lane should feel electrically different from the others. It is the current point of attention. It should feel charged.

### 4. Ensemble rail

This should contain only a few strong global moves.

The ensemble rail is where the performer shapes the whole field:

- build
- drop
- freeze groove
- release
- blackout

This rail should not become a junk drawer. If too many global actions appear here, the system will lose clarity.

### 5. Moment rail

This should remain lightweight but ever-present.

It should allow the performer to:

- capture a moment instantly
- jump back to a prior moment
- compare one moment against another
- branch if desired in a more advanced mode

The moment rail is memory under pressure.

## What The Screen Should Feel Like

The screen should not feel flat.

It should feel like:

- tension can gather
- lanes can thicken
- pressure can move
- phrases can arrive
- instability can spread
- a drop can hit

That means visual language matters enormously.

Possible visual tools:

- brightness shifts for pressure
- blur or smear for instability
- line sharpness for tightness
- width expansion for bloom
- pulse compression for density
- outline glow for focus
- hard snaps on phrase arrival

These should not be decorative effects. They should be semantically meaningful.

## Lane Type Details

The product will get much stronger if each lane type has distinct interaction logic.

### Kick lane

The kick lane is the anchor of force.

It should tend toward:

- high stability
- low melodic variance
- strong phrase authority

Likely signature control:

- `weight`

Likely verbs that matter most:

- `tighten`
- `double`
- `halve`
- `strip`
- `pin`

Likely underlying parameter targets:

- gain
- cutoff
- distortion
- sample choice
- event density
- tail length or clip behavior

The kick should be hard to destroy accidentally. It should feel authoritative.

### Hats lane

The hats lane is the source of chatter, motion, and perceived speed.

It should tend toward:

- high responsiveness
- high density sensitivity
- high phrase shimmer

Likely signature control:

- `chatter`

Likely verbs:

- `scatter`
- `double`
- `tighten`
- `open`
- `twist`

Likely underlying targets:

- density
- velocity patterning
- pan motion
- high-pass
- sample variation
- cut behavior

Hats are one of the easiest places for users to feel expressive quickly. They should reward touch.

### Bass lane

The bass lane is weight, contour, and pressure.

It should tend toward:

- strong coupling with the kick
- medium stability
- high payoff from mutation

Likely signature control:

- `pressure`

Likely verbs:

- `mutate`
- `strip`
- `twist`
- `open`
- `pin`

Likely underlying targets:

- note pattern
- waveform choice
- filter movement
- distortion amount
- octave range
- envelope shape

The bass lane is probably where strong performers will develop a lot of their identity.

### Texture lane

The texture lane is atmosphere, grain, and air.

It should tend toward:

- lower rhythmic authority
- higher timbral elasticity
- stronger bloom behavior

Likely signature control:

- `bloom`

Likely verbs:

- `wake`
- `open`
- `scatter`
- `strip`
- `mutate`

Likely underlying targets:

- room and delay
- filter range
- sample start or end
- noise amount
- motion depth
- stereo movement

The texture lane should be where the product starts to feel cinematic.

### Lead lane

The lead lane is event-like. It should not always be present.

It should tend toward:

- sleeping by default
- high drama when activated
- phrase-oriented entrances

Likely signature control:

- `reach`

Likely verbs:

- `wake`
- `twist`
- `strip`
- `open`
- `pin`

Likely underlying targets:

- note contour
- register
- sustain
- brightness
- delay and room
- melodic variation

The lead should feel special. It should not become constant wallpaper.

## Gesture Details

The exact gestural vocabulary should be treated as product-defining.

### Click

Single click on a lane:

- focuses it
- expands local controls
- makes keyboard actions target it

### Drag across pulse strip

Horizontal drag:

- adjusts density or activity
- previews the effect visually before it lands if quantized

This is likely the first gesture users will learn.

### Hold

Holding a lane for some threshold:

- pins it
- reduces its willingness to mutate
- marks it visually as protected

This is a very "conductor" kind of interaction: hold this line, do not let it drift.

### Upward flick

A quick upward flick could:

- wake a sleeping lane
- increase urgency
- accent the lane for an upcoming phrase

### Downward pull

A downward pull could:

- strip the lane
- reduce density
- pull it backward in the ensemble

### Sweep across multiple lanes

A lateral sweep across several adjacent lanes could:

- apply ensemble pressure
- narrow the field
- temporarily synchronize emphasis

This is where the interface starts to feel bigger than buttons.

## Queue Semantics

The queue should not be a technical afterthought. It should be legible and playable.

Every queued action should communicate:

- what will change
- which target it affects
- when it will land
- whether it can still be cancelled
- whether it depends on another queued action

Example queue entries:

- `bass twist @ next bar`
- `build ensemble @ next 4`
- `wake lead @ phrase boundary`

Advanced behavior could include:

- stacked actions
- action groups
- linked lane and ensemble landings
- "unless cancelled" latching

The queue is one of the main places where intention becomes visible.

## Recovery Design

The product should assume that good performances include mistakes, overreach, and accidental ugliness.

That means recovery is a core design problem.

Recovery tools might include:

- `release`
- `freeze groove`
- `return to moment`
- `pin active anchors`
- `mute unstable lanes`
- `restore last good state for focused lane`

This matters because a performer who trusts recovery can take bigger risks.

Risk and recovery are a pair.

## AI Contract

The AI layer should operate under a strict contract.

It should not be allowed to casually destroy the instrument's coherence.

When invoked, it should respect:

- lane role
- protected anchors
- current tonal center if locked
- phrase structure
- current energy regime
- scope of requested change

Good example requests:

- "mutate the hats without changing their density band"
- "make the bass more rubbery but keep the same rhythmic skeleton"
- "add a sleeping texture voice that can bloom later"

Bad behavior:

- changing every lane when only one was targeted
- changing tempo unexpectedly
- replacing an anchor voice without being asked
- erasing the role identity of a lane

The model should feel daring, but disciplined.

## Mapping To Strudel

The poetic surface needs concrete mappings to Strudel under the hood.

Examples:

`double`
:
- multiply subdivisions
- increase `*N`
- speed a voice pattern
- add alternation or ornament

`halve`
:
- reduce subdivisions
- simplify alternation
- lower event count

`tighten`
:
- shorten tails
- increase `cut`
- reduce room or delay
- reduce swing or looseness

`open`
:
- raise filter ceilings
- widen stereo motion
- increase room and delay
- broaden timbral range

`scatter`
:
- introduce velocity variation
- distribute accents
- apply controlled randomness
- use conditional modifiers

`twist`
:
- change phrasing logic
- alter note order
- swap articulation strategy
- inject an alternate variation path

These mappings should be bounded and role-aware.

## Determinism vs Surprise

Conductor should allow surprise without becoming arbitrary.

This likely means every action should sit somewhere on a predictability spectrum:

- deterministic local controls
- bounded lane verbs
- constrained generative transforms
- rare ensemble surprises

A performer should know the family of likely outcomes even when the exact result is not known.

That is the right balance between trust and discovery.

## Visual Semantics

The visual system should have stable meanings.

For example:

- glow = focused or energized
- pale = suppressed or stripped
- jitter = unstable or mutating
- rigid edges = tightened
- widened geometry = blooming or open
- flashing marker = imminent landing
- locked marker = pinned or protected

The UI should develop a vocabulary that users can learn over time.

## Input Hierarchy

Not all input methods should be equal.

There should be a clear hierarchy:

### Primary

- lane click
- pulse drag
- mute
- solo
- lane verbs
- quantized queueing

### Secondary

- keyboard shortcuts
- command bar
- moment capture

### Tertiary

- code view
- fine-grained parameter inspection
- advanced editing

This keeps the product from collapsing back into an editor.

## Keyboard Grammar

The keyboard layer can be part of the instrument if it is simple and memorable.

Examples:

- `1-9`: focus lanes
- `M`: mute focused lane
- `S`: solo focused lane
- `Q`: cycle quantize target
- `X`: queue mutate on focused lane
- `Z`: queue strip on focused lane
- `D`: queue double
- `H`: queue halve
- `Space`: play or stop
- `Enter`: apply command bar
- `C`: capture moment
- `R`: return to last moment

If this is done well, a skilled user can perform very quickly.

## System Responsiveness

The product has to feel immediate.

That suggests concrete performance constraints:

- local interactions should feel instant
- queue updates should feel instant
- phrase countdown should feel stable
- AI-backed actions should provide immediate pending feedback
- visual state should update before audio changes land

If the system feels laggy or hesitant, it stops feeling like an instrument.

## Audio Truth and UI Truth

One subtle but important principle:

The UI should never confidently claim a musical state that the engine is not actually producing.

This means:

- queued actions should look queued, not completed
- focused edits should preview honestly
- lane state should reflect actual playback conditions
- the UI should degrade gracefully when generation fails

Trust is hard to earn and easy to lose.

## Failure Modes

The vision will fail if Conductor becomes any of the following:

- a beautiful but shallow toy
- a clever prompt wrapper
- a DAW-like UI with AI garnish
- a lane mixer with no real skill ceiling
- a chaotic system with no recovery path

These are useful anti-goals.

## Internal Build Milestones

One way to keep the vision grounded is to define milestones by feel, not just by features.

### Milestone 1

I can focus a lane, drag density, mute and solo, and feel that I am already playing something.

### Milestone 2

Lane verbs feel distinct, role-aware, and worth learning.

### Milestone 3

Phrase-queued changes create visible and audible tension before arrival.

### Milestone 4

Moments make me want to save and return to states during a performance.

### Milestone 5

The system starts to feel semi-alive, with lanes that have inertia and recover differently.

### Milestone 6

Watching another person use it is obviously more interesting than watching a beginner use it.

That last milestone is the real one.
